# Writing Process

Use this process to apply the principles without turning them into a rigid template.

## 1. Identify the project definition

Determine:

- what the project implements;
- the implementation level;
- the language or stack;
- the main task;
- the meaningful distinction from a larger or more complex alternative.

Write this in one or two dense opening sentences.

## 2. Identify the main priority

Find the actual design priority shown by the project, such as readability, minimalism, education, speed, or ease of modification.

Explain how the code or repository structure reflects that priority.

## 3. Find the first meaningful result

Choose the shortest real path that demonstrates the project's central value.

Do not choose a secondary maintenance operation when a direct working example exists.

## 4. Build the first scenario

Collect:

- prerequisites;
- installation or build steps;
- data or model preparation;
- the exact run command;
- the expected effect;
- real output or a success signal.

Keep these pieces close together.

## 5. Add only project-specific sections

After the first result, add only the scenarios the project genuinely supports.

Possible examples include training, inference, tests, benchmarks, quantization, multiple hardware paths, datasets, references, todos, or license.

Do not create a section merely because it is common in generic README templates.

## 6. Remove repetition

Each fact should normally appear once, in the place where it is most useful.

Common repetition to remove:

- describing the project twice in the introduction and TLDR;
- repeating installation in both quick start and installation;
- repeating features in the introduction and a feature list;
- explaining the same architecture in prose and a full repository tree;
- placing limitations both near the command and again in a generic section.

## 7. Verify factual support

Before including any command, number, output, compatibility claim, limitation, or license statement, confirm that it comes from the available project material.

If the evidence is missing, omit the claim or ask for the missing information.

## 8. Finish with a directness pass

Remove:

- generic marketing language;
- corporate phrasing;
- unnecessary background;
- filler transitions;
- claims such as “powerful,” “seamless,” “robust,” or “blazing fast” without concrete support;
- sections with little or no project-specific content.
